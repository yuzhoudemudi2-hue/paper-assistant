# Paper Assistant Requirements Specification

Version: 1.0

Status: Draft

---

# 1. Introduction

## 1.1 Purpose

This document defines the functional and non-functional requirements of Paper Assistant.

Its purpose is to provide a complete specification for developers, AI coding assistants, and future maintainers.

All implementation decisions should satisfy the requirements described in this document.

---

## 1.2 Project Vision

Paper Assistant is an AI-assisted scientific literature monitoring system designed primarily for researchers in theoretical physics.

The system automatically discovers newly submitted papers, evaluates their potential importance, performs AI-assisted analysis, and delivers concise daily reports.

The long-term goal is to reduce the time researchers spend searching for relevant literature while improving awareness of important scientific developments.

---

## 1.3 Objectives

Paper Assistant should:

- Automatically monitor new arXiv submissions.
- Evaluate papers using configurable scientific metrics.
- Assist researchers with AI-generated summaries.
- Provide accurate Chinese translations of scientific content.
- Generate readable daily reports.
- Deliver reports automatically by email.
- Remain reproducible, extensible, and maintainable.

---

## 1.4 Intended Users

Primary users include:

- Theoretical physicists.
- High-energy physics researchers.
- General relativity researchers.
- Graduate students.
- Academic researchers.

Future versions may support broader scientific communities.

---

## 1.5 Scope

The first production version focuses on daily monitoring of newly submitted papers.

The initial scope includes:

- arXiv paper retrieval.
- INSPIRE citation analysis.
- AI-assisted summaries.
- Translation.
- HTML report generation.
- Email delivery.

Future versions may include dashboards, databases, multiple users, and additional scientific resources.

---

## 1.6 Design Principles

The system should emphasize:

- Scientific accuracy.
- Automation.
- Reliability.
- Reproducibility.
- Extensibility.
- Simplicity.

Convenience should never compromise scientific correctness.

---

## 1.7 Definitions

Throughout this document:

Paper

A newly submitted scientific paper.

Author

A researcher associated with a paper.

Citation Count

The total number of citations reported by an external citation database.

AI Analysis

The structured scientific summary generated by an AI model.

Daily Report

The HTML report generated after daily processing.

Pipeline

The ordered sequence of processing stages from paper retrieval to report delivery.

# 2. Overall Description

## 2.1 Product Perspective

Paper Assistant is an independent scientific literature monitoring system.

It integrates multiple external services into a unified automated workflow.

The system is designed around a modular pipeline architecture and should remain independent of any single AI provider or citation database.

---

## 2.2 System Overview

The system performs the following high-level workflow each day:

1. Retrieve newly submitted papers from arXiv.
2. Parse paper metadata.
3. Retrieve author information from INSPIRE.
4. Evaluate paper importance.
5. Perform AI-assisted scientific analysis.
6. Translate selected content into Chinese.
7. Generate daily reports.
8. Deliver reports by email.
9. Archive execution results.

The workflow should execute automatically without manual intervention.

---

## 2.3 Primary Use Case

The primary use case is daily literature monitoring.

A researcher receives a concise report containing the most relevant newly submitted papers without manually browsing arXiv.

The report should help the researcher quickly identify papers deserving further reading.

---

## 2.4 User Characteristics

Typical users are expected to:

- Understand scientific literature.
- Be familiar with arXiv.
- Read English research papers.
- Prefer concise summaries over large volumes of raw information.

The system is intended to reduce information overload rather than replace reading original papers.

---

## 2.5 Operating Environment

The primary operating environment is Windows 11.

Future versions should also support Linux.

The application is expected to run as a scheduled daily task.

No permanent user interaction is required during execution.

---

## 2.6 External Dependencies

The system depends on several external services.

Examples include:

- arXiv
- INSPIRE HEP
- AI providers
- SMTP email servers

The system should tolerate temporary failures of individual services whenever possible.

---

## 2.7 System Boundary

Paper Assistant is responsible for:

- Discovering new papers.
- Collecting metadata.
- Ranking papers.
- AI-assisted summaries.
- Translation.
- Report generation.
- Email delivery.

Paper Assistant is not responsible for:

- Writing scientific papers.
- Verifying scientific correctness of published research.
- Replacing peer review.
- Managing reference libraries.
- Editing PDF documents.

---

## 2.8 Assumptions

The following assumptions are made:

- Internet access is available.
- External APIs are operational.
- Required API credentials are valid.
- The system clock is accurate.
- Scheduled execution occurs once per day.

If these assumptions are violated, the system should fail gracefully.

---

## 2.9 Constraints

The system shall operate within the following constraints:

- Respect external API rate limits.
- Minimize AI usage costs.
- Preserve reproducibility.
- Maintain scientific accuracy.
- Protect sensitive credentials.

Implementation decisions should respect these constraints.

---

## 2.10 Success Criteria

The system is considered successful if it:

- Executes automatically each day.
- Identifies newly submitted papers.
- Correctly ranks papers according to configured rules.
- Produces readable reports.
- Successfully delivers reports by email.
- Operates reliably for long periods with minimal maintenance.

---

## 2.11 Future Expansion

The architecture should support future capabilities including:

- Additional arXiv categories.
- Multiple AI providers.
- Web dashboard.
- Historical database.
- Multiple users.
- REST API.
- Mobile notifications.

These capabilities are outside the scope of the initial release.

# 3. Functional Requirements

## 3.1 Daily Workflow

### FR-001 Automatic Execution

The system shall execute automatically once per day according to the configured schedule.

No manual interaction shall be required during normal operation.

---

### FR-002 Daily Workflow Sequence

The system shall perform the following steps in order:

1. Retrieve newly submitted papers.
2. Parse paper metadata.
3. Collect author information.
4. Evaluate paper importance.
5. Select papers meeting ranking criteria.
6. Generate AI analysis.
7. Translate scientific content.
8. Generate reports.
9. Send reports by email.
10. Record execution logs.

The workflow shall continue whenever possible even if individual components fail.

---

### FR-003 Independent Processing

Each processing stage shall operate independently.

A failure in one stage should not unnecessarily terminate subsequent stages.

Components should report failures through the logging system.

---

### FR-004 Reproducibility

The daily workflow shall produce reproducible results when provided with identical input data and configuration.

Random behavior should be minimized.

---

### FR-005 Configuration Driven

Workflow behavior shall be controlled through configuration files.

Examples include:

- execution schedule
- AI provider
- ranking thresholds
- report language
- email recipients

Changing workflow behavior should not require source code modifications.

---

### FR-006 Execution Summary

At the end of each execution, the system shall generate an execution summary.

The summary should include:

- execution time
- number of papers processed
- number of selected papers
- API failures
- AI requests
- report generation status
- email delivery status

---

### FR-007 Error Recovery

If an external service becomes temporarily unavailable,

the workflow shall continue executing unaffected components whenever practical.

Unavailable information should be clearly identified in the final report.

---

### FR-008 Idempotency

Running the workflow multiple times on the same day shall not create duplicate reports or duplicate notifications unless explicitly configured.

Previously processed data should be reused whenever appropriate.

---
## 3.2 Paper Retrieval

### FR-009 Supported arXiv Categories

The system shall monitor the following arXiv categories by default:

- gr-qc
- hep-th

Additional categories shall be configurable.

Future examples include:

- hep-ph
- hep-lat
- hep-ex
- astro-ph
- quant-ph
- math-ph

---

### FR-010 Daily Submission Detection

The system shall retrieve newly submitted papers from the "recent" section of each configured arXiv category.

Only papers submitted during the current monitoring period shall be processed.

Previously processed papers shall not be processed again unless explicitly requested.

---

### FR-011 Retrieval Frequency

The default execution frequency shall be once per day.

The execution schedule shall be configurable.

The retrieval process shall support manual execution for testing purposes.

---

### FR-012 Metadata Collection

For every retrieved paper, the system shall collect at least the following metadata:

- arXiv ID
- Version
- Title
- Authors
- Abstract
- Subject categories
- Submission date
- Primary category
- PDF URL
- Abstract page URL

Additional metadata may be collected when available.

---

### FR-013 Duplicate Detection

The system shall detect duplicate papers using the arXiv identifier.

Duplicate entries shall not appear in reports.

Previously processed papers shall be recognized automatically.

---

### FR-014 Data Validation

Before further processing, retrieved metadata shall be validated.

The validation shall verify:

- valid arXiv identifier
- non-empty title
- at least one author
- abstract availability
- valid submission date

Invalid papers shall be skipped and recorded in the log.

---

### FR-015 Retrieval Failure Handling

If paper retrieval fails,

the system shall:

- record the failure
- retry according to the configured retry policy
- continue processing other categories whenever possible

The failure shall be included in the execution summary.

---

### FR-016 Category Independence

Each monitored arXiv category shall be processed independently.

Failure in one category shall not prevent processing of the remaining categories.

---

### FR-017 Configurable Categories

Users shall be able to enable or disable monitored categories through configuration files.

No source code modification shall be required.

---

### FR-018 Manual Refresh

The system shall support manually re-running paper retrieval without waiting for the next scheduled execution.

Manual execution shall follow the same validation and logging procedures as automatic execution.

---

### FR-019 Future Compatibility

The retrieval subsystem shall be designed to support future arXiv API changes with minimal modification.

Business logic shall remain independent of transport details.

---
### FR-020 Version Awareness

The system shall recognize different versions of the same arXiv paper.

A newer version of an existing paper shall update the existing record rather than creating a new paper entry.

The system shall preserve:

- original arXiv identifier
- original submission date
- latest version number

Reports shall indicate when a paper is a revised version.

---

### FR-021 Time Zone Consistency

The system shall convert all submission timestamps to a configurable reference time zone before determining daily submissions.

The default reference time zone shall be configurable.

All reports shall display dates and times consistently using the configured time zone.

Internal processing shall use a single reference time zone to avoid duplicate or missing papers caused by time zone differences.

---

### FR-022 Retrieval Audit Trail

The system shall record retrieval metadata for every execution.

The audit record should include:

- execution timestamp
- monitored categories
- number of papers retrieved
- number of new papers
- retrieval duration
- failed categories
- retry attempts

Audit information shall be available for troubleshooting and historical analysis.

---

### FR-023 Future Retrieval Sources

The retrieval subsystem shall be designed to support additional literature sources without modifying the overall workflow.

Potential future sources include:

- OpenAlex
- Semantic Scholar
- Crossref
- INSPIRE literature search

Adding a new retrieval source should require implementing a new retrieval module while preserving the existing pipeline architecture.

---
## 3.3 Metadata Processing

### FR-024 Metadata Normalization

The system shall normalize all retrieved metadata before further processing.

Normalization shall ensure that metadata follows a consistent internal representation regardless of the original source format.

---

### FR-025 Author Name Standardization

Author names shall be standardized before external lookup.

Standardization should preserve the original author names while generating a normalized representation suitable for citation database queries.

The original author list shall remain available for reporting.

---

### FR-026 Author Order Preservation

The original author order shall always be preserved.

Normalization shall never reorder authors.

Reports shall display authors in the original publication order.

---

### FR-027 Duplicate Author Detection

Duplicate author entries within a single paper shall be detected and removed.

Duplicate removal shall preserve the first occurrence.

The original metadata shall remain recoverable.

---

### FR-028 Subject Category Processing

The system shall extract:

- primary category
- cross-listed categories
- complete category list

Category information shall remain available throughout the processing pipeline.

---

### FR-029 Abstract Processing

The abstract shall be normalized before AI analysis.

Normalization may include:

- whitespace normalization
- Unicode normalization
- removal of redundant line breaks

Scientific notation and mathematical expressions shall be preserved whenever possible.

---

### FR-030 LaTeX Preservation

Scientific expressions written using LaTeX syntax shall be preserved.

The system shall avoid modifying mathematical meaning during normalization.

Loss of scientific information shall be minimized.

---

### FR-031 URL Validation

Metadata shall contain valid links for:

- abstract page
- PDF download

Invalid URLs shall be reported through the logging system.

---

### FR-032 Metadata Completeness

The completeness of metadata shall be evaluated.

Missing information shall be recorded.

The absence of optional fields shall not terminate processing.

---

### FR-033 Metadata Quality Score

The system may calculate a metadata quality score.

Factors may include:

- completeness
- consistency
- successful validation
- parsing confidence

Metadata quality shall be available for diagnostics.

---

### FR-034 Persistent Metadata Model

All normalized metadata shall be stored using structured data models.

Business logic shall access metadata through these models rather than raw dictionaries.

---

### FR-035 Metadata Versioning

If metadata changes because a newer paper version becomes available,

the updated metadata shall replace the previous metadata while preserving version history whenever practical.

---

### FR-036 Processing Traceability

Each metadata record shall retain sufficient information to identify:

- retrieval source
- retrieval time
- processing time
- processing version

This information supports reproducibility and debugging.

---

### FR-037 Future Metadata Extensions

The metadata processing subsystem shall support additional metadata fields without requiring redesign.

Examples include:

- DOI
- ORCID
- institution
- funding information
- journal reference
- publication status

Future extensions should preserve backward compatibility.

---
### FR-038 Mathematical Content Preservation

Mathematical expressions appearing in titles and abstracts shall be preserved throughout the processing pipeline.

Normalization shall not remove or alter scientific symbols including:

- Greek letters
- superscripts
- subscripts
- mathematical operators
- tensor notation
- common LaTeX mathematical expressions

Whenever transformations are necessary, semantic equivalence shall be maintained.

---
## 3.4 Author Analysis

### FR-039 Author Lookup

The system shall query author information for every paper using the configured citation database.

The default citation database shall be INSPIRE HEP.

The architecture shall support additional citation providers in the future.

---

### FR-040 Author Matching

Author matching shall maximize accuracy while minimizing false matches.

Matching should consider available metadata including:

- full name
- initials
- ORCID (when available)
- affiliation (when available)

Ambiguous matches shall be clearly identified.

---

### FR-041 Citation Statistics

For each matched author, the system shall retrieve citation statistics including, when available:

- total citations
- number of publications
- h-index

Additional metrics may be collected if supported by the citation provider.

---

### FR-042 Author Profile

The system shall construct a structured author profile.

Each profile may include:

- normalized name
- citation statistics
- publication count
- author identifier
- retrieval timestamp

Author profiles shall be reusable across papers.

---

### FR-043 Multiple Authors

All authors of a paper shall be analyzed independently.

No assumption shall be made regarding the importance of author order.

The complete author list shall contribute to the evaluation process.

---

### FR-044 Citation Cache

Previously retrieved author information shall be cached.

Cached data should be reused whenever valid.

Cache expiration shall be configurable.

---

### FR-045 Lookup Failure Handling

If author information cannot be retrieved,

the system shall:

- record the failure
- continue processing remaining authors
- continue processing remaining papers

Unavailable author statistics shall not terminate the workflow.

---

### FR-046 Missing Citation Information

Authors without available citation information shall remain in the paper metadata.

Missing statistics shall be explicitly represented rather than inferred.

The system shall not estimate missing citation values.

---

### FR-047 Citation Data Validation

Citation statistics shall be validated before use.

Examples include:

- non-negative citation counts
- valid publication counts
- consistent author identifiers

Invalid data shall be ignored and recorded.

---

### FR-048 Author Analysis Traceability

The system shall record:

- citation provider
- retrieval timestamp
- provider response status
- cache status

Traceability information shall support reproducibility and debugging.

---

### FR-049 Future Citation Providers

The author analysis subsystem shall support additional citation databases without modifying business logic.

Examples include:

- OpenAlex
- Semantic Scholar
- Crossref

New providers should implement a common interface.

---

### FR-050 Scientific Integrity

Author analysis shall report retrieved citation statistics only.

The system shall not infer, predict, or fabricate citation metrics.

All reported citation information shall be traceable to the original data source.

---
## 3.5 Paper Ranking

### FR-051 Ranking Objective

The ranking subsystem shall evaluate the potential scientific importance of each paper.

The objective is to assist researchers in prioritizing papers for further reading.

Ranking shall support scientific decision-making rather than replace human judgment.

---

### FR-052 Configurable Ranking Rules

Ranking rules shall be configurable.

Users shall be able to adjust:

- citation thresholds
- scoring weights
- selection thresholds
- ranking strategies

No source code modification shall be required.

---

### FR-053 Citation-Based Evaluation

The ranking subsystem shall evaluate citation statistics collected during author analysis.

Examples include:

- maximum author citation count
- average citation count
- median citation count
- total citation count

Additional statistical indicators may be added in future versions.

---

### FR-054 Maximum Citation Metric

The maximum citation count among all authors shall be calculated.

This metric shall remain available throughout the evaluation process.

---

### FR-055 Average Citation Metric

The average citation count across all matched authors shall be calculated.

Only authors with valid citation information shall contribute to the average.

---

### FR-056 Median Citation Metric

The median citation count shall be calculated whenever sufficient author information is available.

Median statistics shall reduce the influence of extreme outliers.

---

### FR-057 Author Coverage

The ranking subsystem shall record:

- number of matched authors
- number of unmatched authors
- percentage of authors successfully analyzed

Coverage statistics shall be available for diagnostics.

---

### FR-058 Multi-Factor Evaluation

Paper ranking shall support multiple evaluation factors.

Examples include:

- citation statistics
- category
- author analysis
- AI-assisted evaluation
- configurable user preferences

The architecture shall support adding new factors without redesigning the ranking subsystem.

---

### FR-059 AI Evaluation Separation

AI-generated evaluations shall remain independent of objective citation statistics.

Objective metrics and AI assessments shall be distinguishable in reports.

AI-generated opinions shall not overwrite factual data.

---

### FR-060 Ranking Score

Each paper shall receive a final ranking score.

The score shall be reproducible using:

- identical metadata
- identical citation statistics
- identical configuration

The ranking process shall remain deterministic.

---

### FR-061 Ranking Explanation

For every selected paper,

the system shall provide a concise explanation describing why the paper received its ranking.

Examples may include:

- highly cited author
- strong overall author profile
- important research area
- high AI evaluation

Ranking explanations shall improve transparency.

---

### FR-062 Threshold-Based Selection

The system shall support configurable thresholds for selecting papers to include in reports.

Thresholds shall be configurable independently of score calculation.

---

### FR-063 Stable Ranking

If multiple papers receive identical scores,

the ranking order shall remain deterministic.

Tie-breaking rules shall be documented and reproducible.

---

### FR-064 Future Ranking Factors

The ranking subsystem shall support future evaluation factors including:

- institutional reputation
- collaboration network
- publication venue
- citation growth
- user-defined scoring plugins

Future extensions shall not require redesigning the ranking architecture.

---

### FR-065 Ranking Audit

The system shall record:

- ranking version
- configuration version
- scoring components
- final score
- selection decision

Audit information shall support reproducibility and future verification.

---
### FR-066 Highly Cited Author Threshold

The system shall support selecting papers based on configurable author citation thresholds.

The default threshold shall be configurable.

No threshold value shall be hardcoded.

---

### FR-067 Multiple Selection Strategies

The ranking subsystem shall support multiple independent selection strategies.

Examples include:

- at least one highly cited author
- average citation threshold
- maximum citation threshold
- composite ranking score

Users may enable or disable individual strategies through configuration.

---

### FR-068 Scientific Transparency

Every ranking decision shall be explainable.

The report shall identify which evaluation criteria contributed to the paper's selection.

No paper shall appear in the report without an associated explanation.

---

### FR-069 User-Customizable Ranking Profiles

The system shall support multiple ranking profiles.

Examples include:

- Conservative
- Balanced
- Exploratory

Users shall be able to define custom profiles through configuration.

---

### FR-070 Ranking Reproducibility

Given identical:

- papers
- author statistics
- configuration
- ranking algorithm version

the ranking subsystem shall always produce identical results.

Ranking reproducibility shall be considered a core system requirement.

---
## 3.6 AI Analysis

### FR-071 AI Analysis Objective

The AI analysis subsystem shall assist researchers in understanding newly submitted papers.

AI analysis shall complement, rather than replace, reading the original paper.

The generated analysis shall support rapid literature screening.

---

### FR-072 Structured Output

The AI subsystem shall generate a structured analysis for each selected paper.

The analysis should include, when possible:

- research background
- research objective
- methodology
- main contributions
- potential significance
- limitations or open questions

The output structure shall remain consistent across reports.

---

### FR-073 Scientific Accuracy

The AI subsystem shall distinguish between factual information extracted from the paper and interpretations generated by the AI model.

Speculative statements shall be clearly identified.

The AI shall not fabricate scientific claims.

---

### FR-074 Mathematical Content Preservation

Mathematical symbols, equations, and technical terminology shall be preserved whenever practical.

AI prompts shall instruct the model not to simplify or remove important scientific notation.

---

### FR-075 Translation Consistency

Scientific terminology used in AI-generated summaries shall remain consistent with the translation subsystem.

Identical technical terms should be translated consistently throughout a report.

---

### FR-076 Prompt Management

AI prompts shall be stored as external template files.

Prompt templates shall be version-controlled.

Business logic shall not contain embedded prompt text.

---

### FR-077 Prompt Version Recording

The system shall record the prompt version used for every AI analysis.

Prompt version information shall be available for reproducibility.

---

### FR-078 AI Provider Abstraction

The system shall support multiple AI providers through a common interface.

Examples include:

- OpenAI
- Anthropic
- Google Gemini
- DeepSeek

Changing providers shall not require modifications to business logic.

---

### FR-079 AI Failure Handling

If AI analysis fails for an individual paper,

the system shall:

- record the failure
- continue processing remaining papers
- generate the report without terminating the workflow

The report shall clearly indicate unavailable analyses.

---

### FR-080 Token Usage Monitoring

The system shall record AI resource usage.

Examples include:

- prompt tokens
- completion tokens
- estimated cost
- processing time

Usage information shall support optimization and budgeting.

---

### FR-081 Deterministic Configuration

Whenever supported by the selected AI provider,

generation parameters shall be configurable.

Examples include:

- temperature
- top_p
- maximum output length

Configuration shall be recorded for reproducibility.

---

### FR-082 AI Output Validation

Generated content shall be validated before inclusion in reports.

Validation should verify:

- successful completion
- non-empty content
- required sections
- acceptable output length

Invalid responses shall be regenerated or reported.

---

### FR-083 AI Result Traceability

Each AI analysis shall retain:

- provider
- model name
- prompt version
- generation timestamp
- configuration version

Traceability shall support future verification and debugging.

---

### FR-084 Future AI Capabilities

The architecture shall support future AI-assisted features including:

- literature comparison
- novelty assessment
- relationship analysis between papers
- long-term research trend analysis
- interactive question answering

Future capabilities shall not require redesigning the AI subsystem.

---
### FR-085 Scientific Uncertainty

When the AI model is uncertain about an interpretation,

the uncertainty shall be explicitly expressed.

The AI shall prefer stating uncertainty over presenting unsupported conclusions as facts.

---

### FR-086 Reading Recommendation

For each analyzed paper, the AI subsystem shall provide a concise recommendation indicating whether the paper appears to warrant further reading.

The recommendation shall be accompanied by a brief justification based on the paper's content.

The recommendation shall be advisory only and shall not replace the user's scientific judgment.
